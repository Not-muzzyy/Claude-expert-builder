# Knowledge: Open Source Projects

## Why Contribute

- Real-world code exposure — see how professionals structure large projects
- GitHub activity and credibility — merged PRs show up on your profile
- Networking — maintainers and contributors are often hiring or referring
- Learning by reading others' code is underrated

## How to Start

### Step 1: Find a Project
Look for repos with:
- `good first issue` or `beginner friendly` labels
- Active maintainers (recent commits, responses to issues)
- A CONTRIBUTING.md file (means they're organized)

**Where to look:**
- https://goodfirstissue.dev
- https://github.com/explore (filter by language)
- https://up-for-grabs.net
- Projects you already use (Streamlit, Scikit-learn, etc.)

### Step 2: Understand Before Touching
- Read the README and CONTRIBUTING.md fully
- Clone the repo and get it running locally first
- Read existing code in the area you want to change
- Look at past PRs to understand the style expected

### Step 3: Start Small
Good first contributions:
- Fix a typo or documentation error
- Add a missing docstring
- Write a test for an untested function
- Fix a `good first issue` bug

Avoid starting with large feature additions — get a feel for the project first.

### Step 4: Submit a Pull Request
- Fork the repo, create a branch: `fix/issue-123-description`
- Make your change, test it
- Write a clear PR description: what you changed and why
- Reference the issue: `Closes #123`
- Be patient — maintainers are volunteers

## Good Projects for Beginners

| Project | Language | Why |
|---|---|---|
| Scikit-learn | Python | Welcoming community, good docs |
| Streamlit | Python | Relevant to your stack |
| freeCodeCamp | JS / Markdown | Huge, lots of small issues |
| HuggingFace Datasets | Python | ML-relevant, active |
| OWASP projects | Various | Cybersecurity relevant |
| first-contributions | Markdown | Literally made for first timers |

## Turning Your Own Projects Open Source

If you open-source your own work:
- Add a LICENSE file (MIT is the simplest and most permissive)
- Write a clear README (use `templates/repo_readme.md`)
- Add a CONTRIBUTING.md explaining how others can help
- Use GitHub Issues for bug tracking and feature requests
- Respond to issues even if just to say "noted, looking into it"
